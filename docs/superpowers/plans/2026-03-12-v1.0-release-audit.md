# Waaseyaa v1.0 Release Audit Report

**Date:** 2026-03-12
**Auditor:** Claude Opus 4.6 (5 parallel agents + Playwright MCP browser QA)
**Scope:** Full repo (38 packages + 3 metapackages), backend architecture, SSR/Twig, admin SPA, developer experience, browser-driven QA
**Status:** Audit only — no fixes applied

---

## Critical Issues

### CRIT-1. PHPUnit Test Suite Crashes with Fatal Error
- **File:** `tests/Integration/Phase16/NoteApiIntegrationTest.php:218`
- **Error:** `PHP Fatal error: Could not check compatibility between NoteInMemoryStorage::create(): Note and InMemoryEntityStorage::create(): EntityInterface`
- **Root Cause:** `NoteInMemoryStorage` narrows the return type of parent `InMemoryEntityStorage::create()` from `EntityInterface` to `Note`. PHP 8.3+ enforces covariant return types. Additionally, the child completely re-implements storage (maintains separate `$notes` array), making the inheritance pointless.
- **Impact:** **Entire test suite cannot run.** Zero test feedback.
- **Fix:** Either change return type to `EntityInterface` with `@return Note` phpdoc, or implement `EntityStorageInterface` directly without extending `InMemoryEntityStorage`.

### CRIT-2. Foundation (Layer 0) Depends on `waaseyaa/path` (Layer 3) — Architecture Violation
- **File:** `packages/foundation/composer.json`
- **Detail:** `"waaseyaa/path": "@dev"` in `require`. Foundation is layer 0; path is layer 3 (Content Types). This is a 3-layer upward violation.
- **Impact:** Breaks the fundamental layer isolation guarantee. Foundation is imported by nearly every package, creating a transitive dependency on a Content Types package.
- **Fix:** Move whatever Foundation needs from Path into Foundation itself, or restructure the dependency.

### CRIT-3. Validation (Layer 0) Depends on `waaseyaa/entity` (Layer 1) — Architecture Violation
- **File:** `packages/validation/composer.json`
- **Detail:** `"waaseyaa/entity": "@dev"` in `require`. Validation is layer 0; entity is layer 1.
- **Impact:** Layer 0 package importing from layer 1 breaks the downward-only rule.
- **Fix:** Extract entity-specific validation into a higher-layer bridge package.

### CRIT-4. CSRF Blocks All Write Operations — Admin SPA Cannot Create/Edit/Delete Content
- **Reproduction:** Admin SPA → Content → Create → Fill title → Click "Create" → 403 "CSRF token validation failed."
- **Root Cause:** `CsrfMiddleware` (at `packages/access/src/Middleware/CsrfMiddleware.php`) blocks all POST/PUT/PATCH/DELETE requests without a valid session-based CSRF token. The admin SPA (Nuxt on port 3000) makes API calls to the backend (port 8080) but has no mechanism to obtain or send CSRF tokens cross-origin.
- **Impact:** **All content mutations are blocked.** The admin SPA is read-only in practice.
- **Fix:** Either (a) implement a `/api/csrf-token` endpoint the SPA can call, (b) exempt JSON:API requests with `Content-Type: application/vnd.api+json` from CSRF (they're not vulnerable to CSRF attacks), or (c) use SameSite cookie + CORS-based protection instead.
- **Screenshot:** `audit-screenshots/06-admin-csrf-error.png`

### CRIT-5. Stored-XSS via HtmlFormatter — Raw Unsanitized HTML Output
- **File:** `packages/ssr/src/Formatter/HtmlFormatter.php`
- **Detail:** `HtmlFormatter` passes raw entity field content to Twig's `|raw` filter without sanitization. Any HTML stored in an entity field (e.g., via API) is rendered directly, enabling stored XSS attacks.
- **Impact:** Any user who can create/edit content can inject malicious JavaScript that executes for all visitors.
- **Fix:** Sanitize HTML through an allowlist-based sanitizer (e.g., `HTMLPurifier` or Symfony's `HtmlSanitizer`) before passing to `|raw`.

### CRIT-6. No LICENSE File — Blocks Any Public Release
- **Detail:** No LICENSE file exists in the repository root. Without a license, the code is "all rights reserved" by default.
- **Impact:** External contributors cannot legally use, modify, or distribute the framework.
- **Fix:** Add a LICENSE file (e.g., MIT, Apache 2.0, GPL) before any public release.

### CRIT-7. CSRF Blocks MCP Endpoint — AI Agents Cannot Connect
- **Reproduction:** `POST /mcp` with JSON-RPC payload → 403 "CSRF token validation failed."
- **Root Cause:** Same CSRF middleware blocks MCP Streamable HTTP endpoint. MCP clients (Claude Desktop, Cursor) cannot obtain session-based CSRF tokens.
- **Impact:** The entire MCP integration layer is non-functional.
- **Fix:** Exempt `/mcp` from CSRF validation (MCP uses its own auth model).

### CRIT-8. TenantMiddleware Not Wired — Multi-Tenant Isolation Absent
- **File:** `packages/foundation/src/Kernel/HttpKernel.php:135-146`
- **Detail:** `TenantMiddleware`, `TenantContext`, `TenantResolverInterface`, and `NullTenantResolver` all exist in `packages/foundation/src/Tenant/` but are never instantiated or added to the HTTP pipeline. The `Note` entity stores `tenant_id` but nothing enforces it.
- **Impact:** In a multi-tenant deployment, all tenants see each other's data.
- **Fix:** Either wire `TenantMiddleware` into the pipeline and add storage-level filtering, or explicitly document single-tenant-only for v1.0 and remove `tenant_id` from Note.

## High-Priority Issues

### HIGH-1. SSR 403 Forbidden Returns 404 Instead — Security Information Leak
- **File:** `packages/ssr/src/SsrPageHandler.php:139-144`
- **Detail:** When `EditorialVisibilityResolver::canRender()` returns forbidden, the handler calls `renderNotFound()` returning 404. This conflates access control with resource existence.
- **Impact:** Users denied access see "Not Found" instead of "Forbidden", which confirms resource existence by distinguishing 404 from forbidden.
- **Fix:** Return proper 403 with a styled "Forbidden" template.

### HIGH-2. Missing Error Templates (500, 403)
- **Files:** `packages/ssr/templates/` — only `404.html.twig` exists
- **Detail:** No `500.html.twig` or `403.html.twig` templates. `TwigErrorPageRenderer::render()` checks `$this->twig->getLoader()->exists($template)` and returns `null` when missing. Users see raw JSON:API error payloads or unstyled text.
- **Impact:** Poor user experience for server errors and permission denials.
- **Fix:** Create styled 500 and 403 templates. Issue #301 tracks the 403 template.

### HIGH-3. SSR 404 Page Is Unstyled
- **Reproduction:** Navigate to `http://localhost:8080/nonexistent-page`
- **Detail:** Returns bare "Not Found" heading with path text, no navigation, no styling, no "back to home" link. Uses `base.html.twig` layout but the layout itself is minimal.
- **Impact:** Jarring UX for users hitting broken links.
- **Screenshot:** `audit-screenshots/03-ssr-404.png`
- **Fix:** Style the 404 template with site navigation and a home link.

### HIGH-4. Admin SPA Hydration Errors on Every Page
- **Reproduction:** Navigate between any pages in the admin SPA → Console shows hydration warnings/errors on every page load.
- **Detail:** 8 errors + 42 warnings accumulated during rapid navigation through 6 pages. Errors include "Hydration completed but contains mismatches", "Hydration text content mismatch", "Hydration attribute mismatch".
- **Impact:** Vue SSR/CSR mismatch causes DOM inconsistencies. Likely caused by language switcher state (stored client-side) not matching SSR-rendered output.
- **Fix:** Ensure language state is synchronized between Nuxt SSR and client hydration (e.g., via cookie-based locale).

### HIGH-5. Middleware Uses Manual Ordering Instead of Attribute-Based Discovery
- **File:** `packages/foundation/src/Kernel/HttpKernel.php:135-146`
- **Detail:** Middleware ordering is hardcoded via sequential `withMiddleware()` calls: BearerAuth → Session → CSRF → Authorization. The `#[AsMiddleware(priority: N)]` attribute system exists but is unused.
- **Impact:** Adding new middleware requires editing `HttpKernel.php` directly. Contradicts CLAUDE.md documentation.
- **Fix:** Either adopt attribute-based middleware or update docs to reflect manual wiring.

### HIGH-6. Missing Access Policies for menu, path, and relationship Packages
- **Packages:** `packages/menu/`, `packages/path/`, `packages/relationship/`
- **Detail:** No `*AccessPolicy*` files exist. Without access policies, these entity types fall through to deny-by-default in `EntityAccessHandler`, making them inaccessible via API.
- **Impact:** Menu links, path aliases, and relationships cannot be managed via the JSON:API.
- **Fix:** Add access policies for each entity type.

### HIGH-7. SSR Homepage Links to `/admin` Which Returns 404
- **Reproduction:** `http://localhost:8080/` → Click "Admin SPA" card → 404
- **Detail:** The welcome page links to `/admin` but the backend has no route for it. The admin SPA runs on a separate Nuxt dev server (port 3000).
- **Impact:** Broken link on the framework's default landing page.
- **Fix:** Either proxy `/admin` to the Nuxt SPA, or update the link to explain the separate dev server.

### HIGH-8. HttpKernel Has No Top-Level Exception Handler
- **File:** `packages/foundation/src/Kernel/HttpKernel.php:52`
- **Detail:** `handle(): never` wraps routing and auth in try-catch, but `boot()` is unprotected. If boot throws (database missing, config parse error), users see a raw PHP stack trace. `ConsoleKernel` correctly wraps boot.
- **Impact:** Startup failures leak internal details to users.
- **Fix:** Wrap entire `handle()` in try-catch with a generic 500 JSON response fallback.

### HIGH-9. No Admin Authentication/Login UI
- **Detail:** The admin SPA has no login screen, no auth flow, no session management. All admin pages are fully open to anyone who can reach the Nuxt dev server.
- **Impact:** Any anonymous user can view all entity data and attempt mutations (blocked only by CSRF, not by access control).
- **Fix:** Implement authentication UI (login form, session handling) before v1.0.

### HIGH-10. `home.html.twig` Does Not Extend `base.html.twig` — 435 Lines of Inline CSS
- **File:** `packages/ssr/templates/home.html.twig`
- **Detail:** The SSR homepage template is a self-contained 435-line file with all CSS inlined, not extending the base layout. The `entity.html.twig` template renders a bare `<article>` fragment without any layout wrapper.
- **Impact:** No consistent page structure. Entity SSR pages have no head, no navigation, no footer.
- **Fix:** Refactor templates to extend `base.html.twig`.

### HIGH-11. `config()` Twig Function Is Dead Code
- **File:** `packages/ssr/src/Twig/WaaseyaaExtension.php`
- **Detail:** The `config()` Twig function is registered but the config factory closure is never injected — it always returns an empty array.
- **Impact:** Any template using `{{ config('key') }}` silently gets nothing.
- **Fix:** Wire the config factory from the kernel's config loader.

### HIGH-12. 37 of 38 Packages Lack a README
- **Detail:** Only `packages/admin/` has a README.md. All other packages have zero documentation.
- **Impact:** Developers cannot understand what any individual package does without reading source code.
- **Fix:** Add at minimum a one-paragraph README to each package.

### HIGH-13. Twig Template Caching Disabled
- **File:** `packages/ssr/src/ThemeServiceProvider.php:51`
- **Detail:** `'cache' => false` is hardcoded. Every template is parsed from disk on every request.
- **Impact:** Severe performance penalty in production.
- **Fix:** Enable Twig caching with a configurable cache directory.

## Medium/Low Issues

### MED-1. Nav Group Headers Show Machine Names in Admin SPA
- **Reproduction:** Admin SPA sidebar shows "NAV_GROUP_COMMUNITIES", "NAV_GROUP_ELDERS" as section headers
- **Detail:** Nav group labels from the API are not translated or humanized. Core groups ("People", "Content", "Taxonomy") work; custom groups show raw identifiers.
- **Screenshot:** `audit-screenshots/08-admin-french.png`
- **Fix:** Add display labels for nav groups in the entity type API response.

### MED-2. i18n Incomplete for Minoo-Specific Entity Types
- **Reproduction:** Switch to French → Minoo entities ("Resource Person", "Community Group", "Elder Support Request", etc.) remain in English
- **Detail:** Only core framework entity types are translated. Table column headers also remain untranslated.
- **Fix:** Add translations for all entity type labels and field names.

### MED-3. `graphql` Package Is a Shell
- **Package:** `packages/graphql/`
- **Detail:** Has `composer.json` but no `src/` content. Listed as "(stub)" in CLAUDE.md but included in `waaseyaa/full` metapackage dependency tree.
- **Fix:** Either implement or remove from metapackage dependencies.

### MED-4. AI Pipeline Packages Lack Real Integration Testing
- **Packages:** `ai-schema`, `ai-agent`, `ai-pipeline`, `ai-vector`
- **Detail:** Have interfaces, value objects, and executors but `EmbeddingProviderFactory::fromConfig()` relies on external API providers (OpenAI, Ollama). No integration tests validate real API calls.
- **Fix:** Add mock-based integration tests or document AI features as experimental.

### MED-5. Version String Hardcoded as `0.1.0`
- **File:** `packages/foundation/src/Kernel/ConsoleKernel.php:177`
- **Detail:** `AboutCommand` receives `'version' => '0.1.0'` hardcoded.
- **Fix:** Source version from a canonical location for v1.0.

### MED-6. `index.php` Front Controller Has No Boot Error Handling
- **File:** `public/index.php:27-28`
- **Detail:** Only 28 lines, calls `$kernel->handle()` without try-catch.
- **Fix:** Add boot error handling to return a generic 500 response.

### MED-7. Admin SPA Proxy Configuration is Redundant
- **File:** `packages/admin/nuxt.config.ts`
- **Detail:** Both `nitro.devProxy` and `routeRules` forward `/api` to `localhost:8081`. Dual config is redundant.
- **Fix:** Use only one proxy mechanism.

### MED-8. No API Root Index or Discovery Endpoint
- **Reproduction:** `GET /api` → 404; `GET /api/entity-type` → 404; `GET /api/schema` → 404
- **Detail:** No root API endpoint that lists available resources, entity types, or schemas.
- **Fix:** Add a JSON:API root document listing available endpoints.

### LOW-1. Missing `favicon.ico` — Console Error on Every Page Load
- **Reproduction:** Any SSR page → console error "Failed to load resource: 404 (Not Found) @ /favicon.ico"
- **Fix:** Add a favicon.

### LOW-2. Nuxt Admin SPA Has No Custom Error Page (`error.vue`)
- **Detail:** Unmatched routes or server errors fall through to Nuxt defaults.
- **Fix:** Add `error.vue` for consistent error UX.

### LOW-3. Admin SPA Uses Single Browser Target for E2E (Chromium Only)
- **File:** `packages/admin/playwright.config.ts`
- **Fix:** Add Firefox for broader coverage.

### LOW-4. Integration Tests Only Through Phase16
- **Directory:** `tests/Integration/`
- **Detail:** Content types introduced after Phase16 may lack integration coverage.

### LOW-5. Metapackages (`cms`, `core`, `full`) Have No Smoke Tests
- **Detail:** No tests validate that their dependency trees resolve or the combined set boots.

## Architectural Inconsistencies

1. **Layer violations (2):** `foundation→path` (0→3) and `validation→entity` (0→1)
2. **CLAUDE.md says PHP 8.3+, but `composer.json` requires PHP 8.4** (commit `2ac99d00` upgraded to PHP 8.4)
3. **README says admin is "React + Vite"** but it's actually Nuxt 3 + Vue 3
4. **Middleware docs vs reality:** CLAUDE.md documents `#[AsMiddleware]` attribute-based discovery, but actual pipeline uses manual wiring
5. **Tenant system exists as code but is completely unwired** — dead code in a shipping package
6. **`graphql` listed as a package** but is an empty shell
7. **Admin SPA proxy targets port 8081** but the dev server instructions use port 8080

## Recommended Fixes (Priority Order)

### Must Fix Before v1.0 (Blockers)
1. **Fix `NoteInMemoryStorage::create()` return type** to unblock the test suite (CRIT-1)
2. **Exempt JSON:API and MCP from CSRF** — use `Content-Type` or route-based exemption (CRIT-4, CRIT-5)
3. **Resolve layer violations** — remove `waaseyaa/path` from foundation, extract entity dep from validation (CRIT-2, CRIT-3)
4. **Decide tenant strategy** — either wire it end-to-end or remove tenant_id and document single-tenant (CRIT-6)
5. **Add access policies** for menu, path, relationship packages (HIGH-6)
6. **Fix SSR 403→404 conflation** (HIGH-1)
7. **Add boot error handling** to HttpKernel and index.php (HIGH-8, MED-6)

### Should Fix Before v1.0 (Important)
8. **Enable Twig template caching** with configurable directory (HIGH-9)
9. **Create 500 and 403 error templates** (HIGH-2)
10. **Style the 404 page** with navigation (HIGH-3)
11. **Fix admin SPA hydration errors** — likely language cookie sync issue (HIGH-4)
12. **Fix `/admin` link** on SSR homepage (HIGH-7)
13. **Register API routes** for `note`, `vocabulary`, `relationship` entity types (MED-8)
14. **Add `/health` endpoint** for operational monitoring

## Summary

| Severity | Count |
|----------|-------|
| Critical | 8 |
| High | 13 |
| Medium | 8 |
| Low | 5 |
| Missing Features | 14 |
| Architectural Inconsistencies | 7 |
| Documentation Gaps | 8 |
| Playwright QA Failures | 8 of 23 tests |

**Overall Assessment:** The framework has a solid architectural foundation — 36 of 38 packages respect layer discipline, zero dead code, zero TODOs, all files have proper strict types and namespaces. The admin SPA is well-designed with schema-driven forms and good i18n support. However, **8 critical issues block a v1.0 release**: broken test suite, stored-XSS vulnerability, CSRF blocking all write operations (admin SPA and MCP), layer violations, missing LICENSE, and unwired tenant isolation. The stored-XSS (CRIT-5) is the most security-urgent fix; the CSRF issue (CRIT-4) is the highest-impact functional fix since it renders the admin SPA read-only.
